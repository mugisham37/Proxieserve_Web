All states

11 documented in states.html: prompt timeout · declined (mapped reasons) · pay-agent-directly · awaiting agent confirmation · already-paid / nothing due · dispute & refund request · refund issued (channel-aware timing) · connection lost mid-payment (reconcile, don't re-charge) · 3-D Secure failed · fee changed since page open (stale-fee block) · processing (locked + live region).

The cardinal rule across all of them: the user is never left wondering whether money moved. Every terminal state states the charge outcome explicitly.

09 /

Accessibility checklist

Method cards are real radio-equivalent links with visible selection + label
Card fields use inputmode="numeric" + autocomplete="cc-number / cc-exp / cc-csc"
3DS OTP cells labelled aria-label="Digit N"; group has accessible name
Wait-screen progress announced via aria-live="polite" at each step
Processing state uses role="status" + "do not refresh" message
Countdown announced at 30s + 10s thresholds, not every second (no SR spam)
Result heroes (success/fail/refund) announced via role="status" / role="alert"
Colour never sole indicator — every result pairs icon + heading + body text
Touch targets ≥ 44 px; pay CTA is full-width on mobile
Reduced motion: pulse + spinner replaced with static + text per tokens.css
Receipt is real semantic text (not an image) — screen-reader & copy friendly
Amounts always include currency ("RWF 15,000") — never a bare number

Hand-off & self-review

Engineering notes

  • Providers. MTN MoMo Collections API + Airtel Money API for push; a PCI-DSS card processor (Flutterwave / Stripe) for card + 3DS. Card details never touch our servers — tokenized client-side.
  • Idempotency. Every charge carries a client Idempotency-Key; the provider txn reference dedups callbacks. A stale prompt approval after EXPIRED is auto-reversed.
  • Webhooks are the source of truth. The UI's "success" comes from a signed provider webhook, not the client. The wait screen polls status; the webhook flips it. Mismatches favour the webhook.
  • Reconciliation. On reconnect mid-payment, we query the provider with the existing reference rather than re-charging. The "don't pay again — reconciling" state prevents double-payment.
  • Fee re-validation. The amount is re-checked server-side at charge time against the agent-confirmed fee. A changed fee blocks the charge and shows the diff (state 10).
  • Pay-agent. Off-platform payments enter PENDING_AGENT_CONFIRM; the agent confirms receipt in F8, generating the same receipt object. 24h SLA before support escalation.
  • Refunds. Channel-aware: MoMo near-instant, card 5–10 business days. Service fee fully refundable on our error; government fees follow office rules. Refund emits a negative-amount receipt linked to the original.
  • VAT. Computed inclusive and shown as a line on the receipt for tax compliance.
  • Analytics events. pay_method_view, pay_method_select {method}, pay_momo_initiated, pay_momo_approved, pay_momo_timeout, pay_card_initiated, pay_3ds_shown, pay_3ds_result {ok}, pay_success {txn}, pay_failed {reason}, pay_agent_selected, pay_agent_confirmed, pay_dispute_open {reason}, pay_refund_issued, pay_reconcile.
