00 — Foundations

A calm public-service product, dressed in editorial.

Before designing a single screen, this document does three things: analyzes the visual language from your references, extracts a token set that respects ProxiServe's brief (warm, trustworthy, government-adjacent, used on cheap Android phones), and plans the ten-flow delivery.

01 / What the references are actually doing

Your four reference images share a coherent set of moves that I'll lift directly. They also share some agency-website energy that would be wrong for a government-adjacent product — I'll filter those out. Here's the breakdown.

01. Editorial serif as the personality

All headings, nav, button labels, and even body quotes use a high-contrast didone-style serif (Bodoni / Fraunces feel) — frequently in italic. Sans-serif is reserved for utility text, microcopy, and the monospace-like labels in dense rows. This single decision makes the product feel considered rather than templated.

02. The notched corner motif

Multiple references use chamfered / notched corners — opposite-corner notches on color blocks (workplace block), one-corner notch on dark testimonial cards, full-radius pills on buttons. This is the signature we'll carry into ProxiServe as the brand mark for cards, status hero, document slots, and the receipt.

03. Warm cream over flat colour blocks

Backgrounds are a warm desaturated cream (~#F2EBD7), not paper-white. Bold colour blocks (marigold, magenta, kelly green, red, royal blue) sit on top in flat rectangles — never gradient, never glossy. We'll keep the cream foundation everywhere and limit the bold colour blocks to marketing surfaces only.

04. Pill buttons with italic serif labels

White or cream pills with a 1px dark border, italic serif label, optional small arrow glyph. This is the marketing CTA. For dense app surfaces (agent dashboard, admin) we'll mirror this energy with rounded-rectangle sans buttons so density doesn't suffer.

05. Hairline rules + monospace labels

Lists (awards, services) use thin hairline rules with mono-style metadata and pink underlines for inline links. We'll use the same recipe for status timelines, activity logs, audit trails, and pricing rows.

06. Closing gradient band

The "Let's connect" magenta→orange gradient band is the one place gradients are allowed. We'll allow exactly one closing band per long marketing page (landing, service detail, pricing) and never inside the app shell.

What we are not taking from the references

  • Six-up candy-bright service grid — too playful for ID documents
  • Display-size playfulness on form pages — calm wins over clever
  • Bright color blocks behind dashboard chrome — agent + admin stay neutral
  • Decorative line patterns — only used in the landing hero, not elsewhere
  • Italic serif buttons inside dense tables — sans buttons there for speed
  • Agency-style asymmetric nav — we keep nav predictable for low-literacy users

06 / Three zones, one system

ProxiServe lives in three visual zones that share tokens but vary density and chrome.

Public + client

Warm, generous, marketing-quality

  • Cream background everywhere
  • Display serif allowed
  • Pill buttons with italic serif labels
  • Notched color blocks on landing only
  • Generous whitespace (96+ px section padding)
  • Light by default; no dark mode for public landing

Agent

Dense, utilitarian, fast

  • Cream background, same surfaces, no marketing blocks
  • Sans-serif page titles (Fraunces only on key h1)
  • Sans buttons, 10px radius, no italic labels
  • Compact 12px row padding in tables
  • Keyboard shortcuts, dense filters
  • Dark mode supported as user preference

Admin

Most stripped-down, table-heavy

  • Same surfaces; even less decoration
  • No notched corners; flat rectangles + hairlines
  • Audit-first interaction (confirm-by-typing for destructive)
  • Desktop-only for schema builder
  • Dark mode default for power users (toggleable)

Shared across all three

Tokens, primitives, voice

  • Status pills identical everywhere
  • Document Card identical everywhere
  • Form Field component identical everywhere
  • Application code chip identical everywhere
  • Microcopy register identical (plain, second-person)
  • Accessibility floor identical (a11y checklist D.5)

08 / How the three audiences meet

Where client, agent, and admin connect — the seams that have to hold for the value (someone competent will handle this for me) to actually reach the user.

Client → Agent

The application code is the contract

When a client submits via Flow 3, a code (PRX-YYYY-NNNNN) is generated. That same code is the row in the agent's case queue (Flow 8). Every message, every document, every status change flows through this single identifier. The client sees Flow 5's calm status view; the agent sees Flow 8's dense case detail. The data is the same — the chrome is different.

Agent ↔ Client (messaging)

One conversation, two surfaces

The Conversation primitive is the same component in Flow 5 (guest), Flow 6 (authenticated client) and Flow 8 (agent), with role-specific affordances: internal notes only for agents, WhatsApp deep-link only for clients, read receipts on both sides. An agent never sees their internal notes mixed with client messages — visually impossible.

Admin → Service catalogue → Client

Schema is the funnel

When an admin builds a service in Flow 9, the schema renders directly as the dynamic step 2 in Flow 3. Pricing edits with effective dates surface in Flow 1's pricing page and Flow 2's detail page. Service archive states gracefully degrade Flow 3 drafts (schema-mismatch migration).

Admin → Agent

Oversight without surveillance

Admins see all cases in Flow 9 oversight, but with a clear "you are viewing as admin" indicator. Performance dashboards (Flow 8.5 vs 9.2) share metrics — agents see their own, admins see everyone's. Audit log records every privileged action with no ability to delete entries.
