Application submission — edge cases.

The most state-heavy flow in ProxiServe. The 5 live step pages live in the hub. Below are the variants that have to land gracefully — draft resume, validation, offline queue, schema migration, multi-tab conflict, idempotency, draft expiry.

01 / Resume a saved draft

"Pick up where you left off"

Returning

Welcome back, your passport renewal draft is waiting

Last edited yesterday at 18:42 · 3 of 5 steps complete · 4 of 4 required documents uploaded.

Resume →
Mechanics. Draft lives in localStorage for guests; server-side for signed-in users. The banner sits above the top bar on landing, services index, and the service-detail page that matches the draft's service. Dismissable; respects dismissal for 30 days then re-shows.
02 / Photo quality — blurry or dark

"Hard to read — try again?"

Soft warning

This photo is hard to read

It looks dark — your ID number might not be visible. Try again in better light, or use it anyway if you're sure it's clear enough.

Detection. Client-side: brightness histogram + image-laplacian variance to estimate blur. Above thresholds — silent. Below — soft warning, never blocking. Server-side does the same check + OCR confidence; if both fail, the file enters error state.

04 / Sending — button loading state

"Sending your application…"

Submitting
← Back Step 4 of 5 · Review

The form is disabled while sending. Browser back navigation is blocked via beforeunload. A modal blocks the user if they try to close the tab.

Idempotency. A client-generated Idempotency-Key header is included; double-clicks de-duplicate server-side. If the connection drops mid-submit, the retry uses the same key — the server returns the same application code rather than creating a duplicate.
05 / Duplicate click during submit

Toast — "Already sending — hang on"

Idempotent
Already sending — hang on. The first click is on its way. We won't double-submit.
Why this matters. On slow 3G the submit takes 4–8 seconds — users tap multiple times. Without idempotency we'd create duplicate applications. The toast is informational, never alarming.
06 / Offline on submit — queued locally

"Saved on this device — we'll send when you're online"

Offline
!

Saved on this device

Your application is ready to send. We'll dispatch it the moment you're back online — usually within seconds of reconnection. Keep this tab open if you can.

QUEUED · waiting for connection · attempt 1 in 30s
Service worker. The payload is stashed in IndexedDB with a retry schedule (30s, 60s, 5min, 30min). A Background Sync registration handles wake-up. When the submit succeeds, the success toast appears even if the user closed the tab in between — via Push API where granted.
07 / Same draft open in another tab

"This application is open somewhere else"

Conflict

This application is already open in another tab

If you keep editing here, the other tab will become read-only. Otherwise, switch over to that tab — your work there is saved.

Detection. A BroadcastChannel for each draft id; tabs heartbeat every 10s. If the second tab doesn't hear back, it goes read-only with a "your work was paused — refresh to resume" banner. Last-writer-wins is unacceptable for draft applications.
08 / Service schema updated between draft + submit

"We've improved this form"

Migration

We've updated this form since you started

The Department of Immigration changed what they need. Most of your existing answers carry over — please double-check the fields highlighted below:

  • New: "Travel intent" question (we recommend Urgent if < 2 weeks)
  • Changed: Police clearance number format (now starts with RNP-)
  • Removed: "Mother's maiden name" — Immigration dropped this requirement
Versioned schemas. Every service form has a version. On draft resume we compare draft.version vs current.version. Fields that map 1:1 are auto-migrated; new required fields are highlighted yellow; removed fields are silently dropped. A "What changed?" link expands the full diff for the curious.

Soft archive. Drafts of archived services aren't deleted — they migrate to a read-only state. The downloaded JSON is portable for our support team to manually process if needed.
10 / Draft expires in 7 days

"Your saved draft will expire soon"

Expiring

Your draft will expire in 6 days, 14 hours

Drafts are saved on this device for 30 days. After that, we delete it to protect your data. Submit now or save your progress to your account.

Save to account Continue draft
Why 30 days? Compliance: we don't want personally identifying information sitting unencrypted in browser storage indefinitely. The 7-day warning gives users a final nudge — they can extend by creating an account (which moves the draft server-side under their auth).
11 / Server error on submit

"We couldn't send it just now"

5xx

Something on our end went wrong

Your details are safe — nothing was lost. We've logged this for our team to investigate. Most often, retrying in a moment works.

Try sending again Contact support Incident #INC-26-3819
Behaviour. The form remains fully populated. The Retry button uses the same idempotency key. After 3 failed retries we encourage contacting support with the incident ID — which routes directly to the on-call engineer's queue.
12 / Field-level help popover

"What does this mean?"

A11y
Passport number *
Where to find it. Open your current passport to the bio page (the one with your photo). The number is in the top-right corner, near "Passport No." — it starts with letters (e.g. PC).
Implementation. "?" button is a real <button> with aria-describedby linking to the popover content. Keyboard reachable (Tab + Enter). On mobile, tap opens a bottom sheet instead of a popover so it doesn't overlap the field.
