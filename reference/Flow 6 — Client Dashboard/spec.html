Detail tabs are URL-synced query params so a shared link lands on the right tab. [code] is the PRX application code; ownership is enforced server-side — a code you don't own 404s rather than 403s (no existence leak, consistent with F4/F5).

Accessibility checklist

SideNav is a <nav aria-label="Primary">; active link marked aria-current="page"
Mobile tab bar uses role="tablist" equivalent with labelled icon buttons (44 px targets)
Unread counts exposed as text to SR ("Messages, 2 unread"), not colour-only dot
Detail tabs are a real tablist; panels have role="tabpanel" + aria-labelledby
Status hero status announced via role="status"
Conversation thread: each message has author + timestamp in accessible text; system messages distinguished semantically
Optimistic-send pending state announced ("Sending…") + failure announced via role="alert"
Re-auth modal traps focus; Esc disabled (must re-auth) but cancel returns to read-only
Settings toggles are real role="switch" with aria-checked
Danger-zone delete uses confirm-by-typing; button disabled until match
Trusted-device revoke buttons individually labelled with device + location
Colour never sole indicator; reduced-motion + focus-visible inherited from tokens.css

Hand-off & self-review

Engineering notes

  • App shell is the contract. SideNav, UserChip, MobileTabBar, and the content frame are exported from @proxiserve/ui and reused verbatim by F8 (agent) and F9 (admin) — only the nav items and accent differ.
  • Optimistic messaging. Sends render immediately at 60% opacity with a client temp-id; on server ack the bubble solidifies and the temp-id swaps for the real id. On failure → inline retry, message preserved.
  • Conversation parity. Same thread component as F5 (guest) and F8 (agent). Role determines affordances: client never sees agent internal notes; agent sees both lanes visually separated.
  • Offline. Service worker caches the dashboard shell + last-seen app snapshots. Message sends + document uploads queue via Background Sync (shared with F3).
  • Multi-tab. One BroadcastChannel keyed by user id; tabs reconcile unread counts + draft message state so a reply typed in one tab isn't lost in another.
  • Session. 30-day idle for clients (per F4). Expiry mid-session opens a re-auth modal that preserves unsaved work (draft messages, half-filled settings) rather than bouncing to sign-in.
  • Email verification gate. Reads allowed; writes (new application, payment, profile change) blocked behind the verification banner until confirmed.
  • Analytics events. dash_view, dash_banner_action, app_card_click {code}, detail_tab_view {tab}, message_send {optimistic}, message_send_fail, doc_replace {req_id}, settings_section_view, device_revoke, account_delete_attempt, notif_pref_change {channel,on}.
