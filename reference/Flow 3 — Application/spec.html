Step state stored on the server, not in URL

The ?step=N param is a hint for the renderer. Server-side, each draft has furthestStep (the highest step they've completed). Page reloads always land on min(?step, furthestStep + 1) — so refreshing on step 4 never accidentally advances them.

Accessibility checklist

Each step page has unique <title> ("Step N of 5 — [step] · [service]")
aria-current="step" on active stepper item
Live region announces step change and save-state changes
All actions keyboard-reachable; sticky footer never traps focus
Field errors associated via aria-invalid + aria-describedby
Help popovers reachable by keyboard; Esc closes
Touch targets ≥ 44 px — radio-cards, dropzone buttons, file actions
Drop targets keyboard-actionable; status announced via aria-live
Camera capture: accept="image/*" capture="environment" on mobile
Final consent checkboxes individually labelled (not grouped)
Application code in confirmation announced via role="status"
Copy-to-clipboard announces "Code copied to clipboard"
Color never sole indicator — every status carries icon + text
Reduced motion respected globally via tokens.css
No JS fallback: server-rendered form with <noscript> instructions to enable JS

Hand-off & self-review

Engineering notes

  • Schema-driven step 2. Service schemas live in admin (Flow 9.3). Form rendering uses a generic <SchemaRenderer schema={…} value={…} onChange={…}> — same component used in admin's live preview.
  • Auto-save. Debounced 1.5 s after last edit. PUT /api/drafts/[draftId] with the full form state. Status surfaced via the AutoSaveIndicator in the top bar.
  • Idempotency. Client generates Idempotency-Key: uuid-v4 per draft. Server stores; replays return the same response within 24 hours.
  • File uploads. Resumable (tus protocol) for files > 1 MB. PUT directly to object storage with signed URLs — never through the app server. Virus scan runs asynchronously; client polls for status.
  • Offline. Service worker registers Background Sync for the submit endpoint. Files stashed in IndexedDB until upload window opens.
  • Multi-tab. One BroadcastChannel per draft id. 10s heartbeat. Newcomer wins; loser goes read-only.
  • Schema migration. Each draft stores schemaVersion. On load, fetch current schema; if mismatch, run migrate(draft.value, oldSchema, newSchema) — yields {migratedValue, additions, removals}. UI shows diff.
  • Photo quality. Client-side: brightness mean < 60 OR laplacian variance < 100 triggers soft warning. Server: OCR confidence < 0.7 triggers backend warning to agent in Flow 8.
  • Analytics events. apply_start, apply_step_view {n}, apply_step_complete {n}, apply_save_exit {step}, apply_resume, apply_doc_upload {req_id, type}, apply_doc_error {reason}, apply_submit_attempt, apply_submit_success {application_code}, apply_submit_error {reason}, apply_account_prompt_action.
