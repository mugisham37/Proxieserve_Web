Auth — edge cases.

Every defensive UX an auth flow needs: rate-limited lockouts, "did you mean to sign in?" hints, account-exists nudges on signup, "new device" alerts, OTP failure paths, magic-link sent confirmations.

01 / Invalid credentials — wrong password

"That doesn't look right"

4xx

Welcome back

Sign in to manage your applications.

That email and password don't match. Try again, or use "forgot password" if you can't remember.
Password Forgot?
Tone. Never expose which field is wrong (preserves account-existence privacy). Form keeps the email value but clears password. After 3 attempts a soft captcha appears; after 5 we lock for 5 min — see state 02.
02 / Locked — too many attempts

"Take a short break"

Rate-limited

Locked for safety

Too many sign-in attempts. We've paused new tries to protect your account.

04:38 UNLOCK IN
Forgot it? You can reset your password — that bypasses the lockout.
Reset password instead →
Logic. Sliding window: 5 failed attempts in 10 minutes triggers a 5-minute lockout. The countdown is per-IP & per-account. Password reset uses a separate channel (email or SMS), so it's always available.
03 / Did you mean to sign in?

"This email already has an account"

Hint

Create your account

We've seen this email before. Sign in instead?
Friendly redirect. On signup, if the email matches an existing account, we don't error — we show the hint and let them switch to login if they want. Trying to create the duplicate fails server-side, but the hint catches most cases first.

Behaviour. Highlight all 6 cells in error (the user can't know which is wrong). Clear on focus. After 3 wrong attempts the code is invalidated and the user must resend.
05 / OTP — expired

"This code has expired"

Expired

Enter the code

Sent to aline.m@example.com.

This code has expired. Codes are only valid for 5 minutes. Request a new one.
Don't punish the user. Network delays cause this often. Resend has no cooldown after expiry. Each new code invalidates the old one — no race conditions.

Triggers. First sign-in from new browser/IP combo. Also fires on geo jumps. After verification, the device is trusted for 30 days unless the user revokes from settings.
07 / Reset link sent

"Check your email"

Sent

Check your inbox

We sent a reset link to:

aline.m@example.com

Link expires in 15 minutes. Don't see it? Check spam, or resend.

Privacy by default. The page says the same thing whether the email matches an account or not — never confirms account existence to enumeration attackers. Internally, we only send the email if the address is registered.
08 / Reset link expired or used

"This link's used up"

Expired

Link no longer works

Reset links can only be used once and expire after 15 minutes — whichever comes first.

What now? Request a new link. It'll arrive in seconds.
Send a new reset link → Or sign in normally
Compromise window. One-time tokens + short expiry mean a leaked email can't lead to account takeover later. We log every reset attempt — admin sees patterns.
09 / Claim — code doesn't exist

"We couldn't find that code"

Not found

Link your guest application

Add your tracking code to manage your application from your account.

We couldn't find a tracking code matching that. Double-check the dashes and digits.
Codes look like PRX-2026-NNNNN — found in the SMS we sent when you applied.
Don't over-share. We say "we couldn't find it" — never "this code belongs to someone else". If the code exists but belongs to another account, we still show the not-found message; the rightful owner can support to merge.
10 / Email verification banner

"Please confirm your email"

Verification
Just one thing — we've sent a confirmation link to aline.m@example.com. Click it to unlock account features.
Where it appears. Persistent banner above the dashboard until verified. Account works (can view applications), but writes (new applications, payments) are blocked until email is confirmed. The banner is dismissible for the session.
