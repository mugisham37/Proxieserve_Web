URL params (shareable deep links)

  • q — search string (debounced 200 ms, URL-synced on blur).
  • category — single-select enum (tax, identity, business, welfare, permits, other).
  • sort — enum (popular, eta, fee, az).

Accessibility checklist

Filter chips are <button aria-pressed>
Search input has role="searchbox" and visually-hidden label
Results count announced via aria-live="polite"
Keyboard: "/" focuses search; "Esc" clears + blurs
Service cards are whole-card links; inner pseudo-CTA aria-hidden so SR doesn't double-announce
Category dot is decorative — accompanied by visible category label text
Sub-nav uses aria-label="On this page" and tracks scroll position
Action card has its own aria-label ("Start this service")
Requirement help "?" buttons use aria-describedby to expose the tooltip text
Eligibility gate is a focus-trapped modal; Esc closes
Sticky mobile bar has aria-label="Start application for [service name]"
Touch targets ≥ 44 × 44 px (filter chips have 12 px vertical padding)
Colour never sole indicator — every status pill includes icon + text label
Reduced motion respected via tokens.css global rule
Sortable table column not used — sort is a single-select for clarity

Hand-off & self-review

Engineering notes

  • Schema source. Service definitions live in the admin (Flow 9.3). Each service has: slug, name {en, rw, fr}, category, description, fee, urgentFee, eta {min,max}, requirements[], flags: {new, pilot, archived, waitlisted, requiresInPerson, multiApplicant}, eligibilityChecks[].
  • Filtering. All filters client-side (catalogue is bounded ~30 services). URL-synced. Server only renders the initial set.
  • Empty / archived / waitlisted are not separate routes — they're variant renders driven by service flags.
  • Sub-nav scroll-spy uses IntersectionObserver; falls back to scroll listener on iOS < 12.4.
  • Sticky action card uses native position: sticky; no JS scroll math.
  • SEO. One indexed canonical per service slug. Filtered URLs return <link rel="canonical"> back to /services. Service detail emits JSON-LD Service + Offer schema.
  • Analytics events. services_index_view, services_search {q}, services_filter {category}, services_sort {sort}, service_card_click {service_id}, services_empty_state, service_detail_view {service_id}, service_start_click {service_id}, service_save_for_later {service_id}, service_archived_view, service_waitlist_join, service_request {term}.

Designer self-review rubric (D.8)

01 — Real content (every service has real name, fee, ETA, description)
02 — Spacing scale obeyed
03 — Semantic tokens only
04 — Mobile-first (320 viewport renders without scroll, sticky mobile CTA works)
05 — Sticky / safe-area respected
06 — All states covered (10 in states.html)
07 — Reduced-motion variant inherited from tokens.css
08 — A11y checklist signed (15/15)
09 — i18n length budget (chips and card descriptions tested at +35%)
10 — Icon family consistent (Lucide; one magnifier + one chevron)
11 — Banned patterns absent
12 — Hierarchy obvious
13 — CTAs unambiguous ("Start application", "Save for later", "Join waitlist", "Notify me", "Request a service")
14 — Edge cases (10 documented: archived, waitlisted, pilot, eligibility, in-person, multi-applicant, empty search, loading, 404, offline)
